# 디자이너를 위한 개발 개념 정리

개발하면서 자주 마주치는데,
디자이너 입장에서는 처음엔 헷갈릴 수 있는 개념들을 정리했어요.

짧게 보면,
이 문서는 **개발 환경 / 권한 / 설치 / 배포 전 개념**을 이해하기 위한 메모예요.

---

## 1. Bun

**Bun은 자바스크립트/타입스크립트 실행 도구이자 패키지 매니저예요.**

쉽게 말하면,
`npm`, `node`, `npx` 같은 역할을 일부 같이 해주는 도구예요.

왜 나오냐면,
요즘 어떤 툴이나 플러그인은 `bun`을 기준으로 돌아가요.
특히 Claude Code 채널 플러그인 문서에서 Bun이 필요했어요.

디자이너 감각으로 비유하면,
**Figma 파일을 열기 위한 전용 앱 같은 것**에 가까워요.
파일이 있어도,
그걸 실행해줄 런타임이 없으면 못 여는 거예요.

### 기억할 점

- `bun`이 없으면 Bun 기반 플러그인/스크립트는 실행이 안 돼요.
- 설치하려면 시스템 권한이 필요한 경우가 있어요.
- 프로젝트마다 필요한 실행 도구가 다를 수 있어요.

---

## 2. sudo

**sudo는 관리자 권한을 잠깐 빌려서 명령을 실행하는 방식이에요.**

리눅스나 서버 환경에서는,
모든 사용자가 시스템 전체를 수정할 수 없어요.
그걸 막아두는 게 기본이에요.

그래서 일반 사용자로는 안 되는 작업을 할 때,
`sudo`를 붙여서 관리자 권한으로 실행해요.

예:

```bash
sudo apt-get install unzip
```

이건
`unzip`을 설치할 권한을 잠깐 빌려 실행하는 거예요.

### 왜 필요했나

우리가 Bun을 설치하려고 했는데,
설치 과정에서 `unzip`이 필요했어요.
근데 `unzip`을 설치하려면 시스템 영역을 건드려야 해서,
관리자 권한이 필요했어요.

그런데 그 환경엔 `sudo`도 없고,
root 권한도 없어서 설치가 막혔어요.

### 기억할 점

- `sudo`는 “권한을 올리는 명령”이에요.
- `sudo`가 없으면 시스템 설치 작업이 막힐 수 있어요.
- 문제가 명령어가 아니라 **권한 구조**일 때가 많아요.

---

## 3. root 권한

**root는 시스템 최고 관리자예요.**

디자인 툴로 비유하면,
편집 권한이 있는 사람이 아니라
**워크스페이스 소유자**에 가까워요.

일반 사용자는 자기 폴더는 만질 수 있어도,
시스템 전체는 못 만져요.
root는 그걸 다 할 수 있어요.

우리가 겪은 문제는,
`claworc` 사용자로는 시스템 패키지를 설치할 수 없었고,
root나 sudo가 없어서 거기서 멈춘 거예요.

---

## 4. apt-get

**apt-get은 리눅스에서 패키지를 설치하는 명령이에요.**

맥의 Homebrew,
모바일의 앱스토어,
Figma 플러그인 설치와 비슷하게 생각해도 돼요.

예:

```bash
apt-get install unzip
```

이건 시스템에 `unzip`이라는 프로그램을 설치하라는 뜻이에요.

다만,
이건 시스템 영역을 바꾸는 작업이라 보통 관리자 권한이 필요해요.

---

## 5. unzip

**zip 압축을 푸는 도구예요.**

Bun 설치 스크립트가 내부적으로 압축 파일을 받아서 푸는 과정이 있었고,
그때 `unzip`이 없어서 설치가 중간에 멈췄어요.

즉 이건 큰 개념이라기보다,
설치에 필요한 작은 의존성 중 하나였어요.

---

## 6. 권한 문제 vs 설치 문제

이건 구분하는 게 중요해요.

### 설치 문제가 아닌 경우

- 명령은 맞음
- 필요한 패키지도 맞음
- 그런데 `Permission denied`가 뜸

이럴 땐 보통 설치 방법 문제가 아니라,
**권한 문제**예요.

이번 상황이 딱 그랬어요.

### 설치 문제인 경우

- 명령이 틀림
- 패키지 이름이 틀림
- 파일 경로가 틀림
- 툴 버전이 안 맞음

---

## 7. CLI

**CLI는 명령줄 인터페이스예요.**

버튼 누르는 UI 대신,
텍스트 명령으로 툴을 다루는 방식이에요.

예:

```bash
claude --help
```

이건 Claude Code를 CLI로 다루는 예예요.

디자이너 입장에서는 처음엔 낯설 수 있는데,
반복 작업이나 자동화에 강해요.

---

## 8. 플러그인과 스킬 차이

이건 Claude Code 쓰면서 특히 헷갈리기 쉬워요.

### 플러그인

**기능을 확장하는 설치물**이에요.

예:
- frontend-design
- telegram channel plugin
- figma plugin

플러그인은 Claude Code 안에
새 명령, 새 연동, 새 기능을 붙여줘요.

### 스킬

**일하는 방식이나 지침 묶음**이에요.

예:
- 어떤 프로젝트를 어떻게 계획할지
- 어떤 형식으로 문서를 만들지
- 특정 작업에서 어떤 절차를 따를지

### 한 줄 요약

- 플러그인 = 기능 추가
- 스킬 = 작업 방식 추가

---

## 9. OMC(oh-my-claudecode)

**Claude Code를 더 잘 쓰기 위한 확장 레이어**예요.

팀 모드,
에이전트 오케스트레이션,
작업용 스킬,
플러그인 기반 워크플로우를 더 쉽게 쓰게 해줘요.

우리가 한 건 대략 이거예요.

- OMC 설치
- 팀 실행 확인
- 프로젝트별 `CLAUDE.md` 정리
- frontend-design 플러그인 확인/설치

즉 Claude Code를 그냥 쓰는 게 아니라,
좀 더 조직적으로 쓰게 만드는 도구라고 보면 돼요.

---

## 10. CLAUDE.md

**Claude Code가 프로젝트를 이해할 때 먼저 읽는 프로젝트 지침서**예요.

이건 디자이너 입장에서 꽤 중요해요.
왜냐면 이 파일에
디자인 톤,
기술 스택,
하지 말아야 할 것,
우선순위를 적어두면,
Claude가 훨씬 덜 엉뚱하게 움직여요.

예를 들면,
이 프로젝트에서는 이런 걸 적어둘 수 있어요.

- 바닐라 HTML/CSS/JS로 간다
- fig000 같은 에디토리얼 톤을 참고한다
- 불필요한 프레임워크는 넣지 않는다
- 모바일 2열, 웹 5열

즉,
**내 의도를 Claude가 기억하게 만드는 프로젝트 브리프**예요.

## 11. Git / GitHub / Push

### Git

로컬에서 버전을 관리하는 도구예요.
수정 이력을 남기고,
변경을 묶어 관리해요.

### GitHub

그 Git 저장소를 온라인에 올려두는 곳이에요.
협업하거나 배포 연결할 때 많이 써요.

### Push

로컬에서 만든 변경사항을 GitHub로 올리는 거예요.

예:

```bash
git push
```

즉,
- Git = 로컬 이력 관리
- GitHub = 온라인 저장소
- Push = 업로드

## 12. 프로젝트 레포

**레포(repository)**는 프로젝트 폴더이자 버전 관리 단위예요.

우리가 새로 만든 `web-build-project`도 레포예요.
레포를 새로 만든다는 건,
그 프로젝트를 독립된 작업 단위로 관리하겠다는 뜻이에요.

이건 꽤 중요해요.
왜냐면 프로젝트를 분리해두면
맥락이 덜 섞이고,
Claude도 덜 헷갈려요.

---

## 13. 지금까지 겪은 걸 한 줄씩 다시 정리하면

- `bun`은 실행/설치용 도구였어요.
- `sudo`는 관리자 권한을 빌리는 명령이었어요.
- `apt-get`은 시스템 패키지 설치 명령이었어요.
- 이번 문제는 설치 방법보다 권한 문제였어요.
- `frontend-design`은 스킬이 아니라 플러그인이었어요.
- `CLAUDE.md`는 프로젝트용 작업 지침서였어요.
- OMC는 Claude Code를 더 구조적으로 쓰게 해주는 레이어였어요.

---

## 14. 디자이너 입장에서 특히 기억하면 좋은 것

이것만 먼저 기억해도 충분해요.

1. **안 되는 이유가 코드가 아니라 권한일 수 있다**
2. **플러그인과 스킬은 다르다**
3. **프로젝트마다 Claude에게 브리프를 줘야 한다**
4. **레포를 분리하면 작업이 훨씬 덜 헷갈린다**
5. **CLI는 낯설지만 자동화에 강하다**
